Why It Matters

The business case for live streaming responses

😊 Higher customer satisfaction

Answers appear immediately and grow token-by-token, so the wait feels far shorter even when the total time is unchanged. Users stay engaged instead of staring at a spinner.

🔍 Transparency builds trust

The inline thought process reveals what the agent is doing behind the scenes — searching, planning, analyzing — so the interaction feels meaningful and the answer feels earned, not guessed.

📉 Fewer abandoned conversations

Continuous, visible progress keeps people from giving up on longer answers. They see momentum, trust it is working, and wait for the full result.

⭐ Meets modern expectations

Users now expect the ChatGPT / Copilot style of live reply. A static "pause, then dump a wall of text" response feels dated and lowers confidence in the agent.

Streaming is not a UI gimmick — it is a measurable lever on satisfaction, trust, and completion, and the reason this playground was worth building.
The Goal

Render a real, progressive streaming answer

We wanted the same experience users expect from modern AI chat:

  • ①A single answer bubble that grows token-by-token in place.
  • ②An inline "thought process" showing the agent's reasoning steps while it works.
  • ③The view auto-following the answer as it streams.
  • ④A diagnostics inspector to see raw streaming activities.

The stack

Web Chat: CDN botframework-webchat/latest

Client: agents-copilotstudio-client (Direct-to-Engine)

Auth: @azure/msal-browser (Entra redirect)

Relay: Express token server, port 3978

Streaming "just working" out of the box is a myth — Web Chat and the SDK each make assumptions you have to actively reconcile.

The Good News

What's working ✅

✅ Token-by-token streaming (DtE and DL)

Both transports stream: the Direct-to-Engine SDK natively, and Direct Line once the client sends deliveryMode:"stream". Same informative → streaming → final chunks either way.

✅ Single growing bubble

Forcing the activity id = streamId makes Web Chat update one bubble in place — verified growing 8 → 1385 chars in a single message.

✅ Dynamic "thought process"

The informative chunks are real reasoning steps. We render them as a styled timeline that disappears the moment the answer streams.

✅ Auto-scroll & diagnostics

A MutationObserver keeps the view pinned to the growing answer. The inspector confirmed 439 chunks across 2 livestreams end-to-end.

With the right normalization, the full modern streaming UX is achievable — single bubble, inline reasoning, live scroll.
The Hard Parts

What's not working (out of the box) ⚠️

⚠️ 152 bubbles, not 1

Web Chat dedupes by activity id. Since every chunk has a unique id, each renders as its own bubble. Long answers exploded into 150+ stacked bubbles.

⚠️ Cold observable, one subscriber

The SDK's activity$ is cold with a single internal subscriber. Subscribe twice (e.g. for an inspector) and one observer goes blind.

⚠️ No auto-scroll on in-place updates

Web Chat only auto-pins on new activities. A streaming answer updates the same id, so the view stays put while content grows off-screen.

⚠️ Direct Line needs an explicit opt-in

Plain Web Chat over Direct Line paints only the final message — until the client sends deliveryMode:"stream". Miss the opt-in and you'll wrongly conclude DL can't stream (we did, at first).

None of these are bugs exactly — they're mismatched assumptions between the SDK's streaming model and Web Chat's transcript model.
The Fixes

How we reconciled them

  • ▸Stamp id = streamId on every interim chunk so Web Chat groups them into one growing message (also set from.role = "bot").
  • ▸Single subscription + tap pattern — wrap the SDK connection once, then tap each activity for both the inspector and Web Chat. Never subscribe to activity$ twice.
  • ▸MutationObserver auto-scroll — watch the transcript DOM and snap to bottom while the user is near the bottom (respects manual scroll-up).
  • ▸Withhold informative chunks from Web Chat — route them to the custom "thought process" card instead, then fade it when streaming begins.
// The one-line fix that collapsed 152 bubbles → 1
function normalizeStreamingForWebChat(activity) {
  const info = getStreamInfo(activity);
  if (info?.streamId) activity.id = info.streamId;   // group by stream, not chunk
  return activity;
}
Every fix lives in a thin normalization layer between the SDK connection and Web Chat — no forks, no patched libraries.
Two Workable Paths

Live streaming works on both transports

✅ Direct-to-Engine (DtE) SDK

Native streaming. CopilotStudioWebChat.createConnection(client, { showTyping:true }) emits generative chunks with no extra opt-in. Entra (MSAL) sign-in; no secret in the browser.

Best for: signed-in users, richest reasoning steps, zero secret exposure.

✅ Direct Line (DL) + opt-in

Streams once you ask for it. Standard Web Chat over Direct Line paints only the final message until the client signals deliveryMode:"stream" — then the same streamType:streaming chunks flow over the WebSocket.

⚠️ Agent prerequisite: the agent flag Channels_SupportForStreamingEnabled must be ON, or Direct Line never emits streaming chunks no matter what the client sends.

Best for: anonymous / token-endpoint embedding, parity with the official test canvas.

// PREREQUISITE: enable the agent flag Channels_SupportForStreamingEnabled
// DL opt-in — exactly what the official test canvas sends
// 1) once, when the connection goes online:
directLine.postActivity({ type:"event", name:"startConversation",
  deliveryMode:"stream",
  entities:[{ type:"ClientCapabilities", requiresBotState:true,
              supportsListening:true, supportsTts:true }] });
// 2) on every outgoing user message:
postActivity({ ...activity, deliveryMode:"stream" });   // 0 → 84 chunks
Verified against the same agent: DtE streams natively; DL streams the instant you send deliveryMode:"stream" (with Channels_SupportForStreamingEnabled ON) — confirmed 1 livestream / 84 chunks / 1 finalized.

Field Notes

Smaller gotchas worth knowing

  • •Web Chat sorts the transcript by webchat:sequence-id — get this wrong and bubbles render out of order.
  • •net::ERR_ABORTED on the conversations POST is benign — it's the SSE stream closing.
  • •Token is cached by MSAL, so reconnects are silent (no second sign-in).
  • •Informative chunks carry a lower sequence-id than streaming chunks — they genuinely precede the answer.
  • •Browser caching bites hard during dev — hard-reload after edits.
  • •showTyping: true on the connection polishes the latency loader UX.
Budget time for the plumbing, not the UI. The visible chat is easy; the activity reconciliation is where the work is.
